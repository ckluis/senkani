Hook Architecture All Agents

Senkani registers a compiled binary (~/.senkani/bin/senkani-hook) as a PreToolUse and PostToolUse handler in your agent's settings. The binary speaks JSON over stdin/stdout and has zero non-Foundation dependencies — it starts in under 5ms.

PreToolUse fires before

Fires before any registered tool executes. Can allow the call (passthrough), redirect it to a senkani MCP tool, or deny it with a reason message that the agent receives instead of a tool result.

Input: JSON object with hook, tool_name, and tool_input fields.
Output: {} (passthrough) or {"hookSpecificOutput": {"permissionDecision": "deny", "permissionDecisionReason": "..."}}.
Latency budget: <5ms — must be imperceptible. Daemon connection is via Unix socket at ~/.senkani/hook.sock; 5ms response timeout after which hook returns passthrough automatically.

PostToolUse fires after

Fires after a tool completes. Used for metrics recording, cache warming, incremental index updates, and triggering async auto-validation. No latency constraint — the tool has already run.

Input: Same schema as PreToolUse, plus tool_result.
Use cases: Write/Edit completions trigger incremental symbol re-index for the touched path; cache entries for the path are invalidated; token savings are recorded to the session database.

Routing Logic

On each PreToolUse event, HookRouter decides whether to redirect to a compressed MCP tool, pass through unmodified, or deny. Mutating commands always pass through — Senkani never interferes with writes.

Fail-Safe Design

The hook binary is designed to fail open. Any error — socket not found, daemon slow, malformed JSON, process crash — results in an empty passthrough response. Your agent session is never blocked by Senkani.

✓ Fail-safe guarantee: Socket unreachable → passthrough (<1ms). Daemon response timeout (5ms) → passthrough. Non-zero exit or malformed output → passthrough. A Senkani crash cannot block your agent.
settings.json registration

senkani init writes the following to ~/.claude/settings.json (or the equivalent for your agent). It merges with existing config — running it twice is safe.

{ "hooks": { "PreToolUse": [{ // fires before tool executes — can allow, deny, or redirect "matcher": "Read|Bash|Grep|Write|Edit", "hooks": [{ "type": "command", "command": "~/.senkani/bin/senkani-hook" }] }], "PostToolUse": [{ // fires after tool completes — metrics, cache, index updates "matcher": "Read|Bash|Grep|Write|Edit", "hooks": [{ "type": "command", "command": "~/.senkani/bin/senkani-hook" }] }] } }
ℹ Note on matchers: Never use an empty matcher string — it would intercept every tool call in every project globally. Senkani uses Read|Bash|Grep|Write|Edit explicitly. senkani doctor flags misconfigured matchers.
Multi-agent compatibility

The hook binary is registered globally in ~/.claude/settings.json and is invoked by the MCP runtime — it doesn't care which AI triggered the call. Cursor, Copilot agent mode, and Codex all respect PreToolUse/PostToolUse hooks at varying fidelity levels.

Tier 1 — Claude Code (100%): All hook events, Layer 3 smart denials, full budget enforcement, direct session JSONL for exact token counts.
Tier 2 — Cursor / Copilot / Codex (~90%): Read/Bash/Grep interception, budget tracking, Layer 3 patterns all active. Deny response handling varies by host.
Tier 3 — Any MCP agent (~40%): 13 senkani tools callable directly without hook registration.

⚠ For Tier 2 agents: senkani init --hooks-only writes just the hooks config and skips Claude Code-specific MCP registration. Run senkani doctor to verify hook delivery for your specific agent.

Filter Pipeline

Shell output is compressed before the AI reads it. senkani_exec runs a command and applies a rule set to the output — 24 built-in rules covering the noisiest commands in a typical dev workflow.

FilterEngine operations

Rules are matched by (command, subcommand) pair and applied as an ordered chain. Each operation is composable — a single rule can chain several.

stripANSI — remove all ANSI escape sequences and control characters
head(n) / tail(n) — keep first or last n lines; discard the rest
truncateBytes(n) — hard byte cap on total output
keepMatching(pattern) — discard lines not matching a regex
stripMatching(pattern) — discard lines matching a regex
dedupLines — collapse consecutive identical lines into one + count
groupSimilar — fold runs of similar lines into "… and N more"
stripBlankRuns — collapse multiple blank lines to one

Mutating commands always pass through

Commands that write to disk or have side effects are never intercepted by the filter pipeline. Senkani only filters read-only output — it never touches your git history, filesystem mutations, or network calls.

Passthrough allowlist (always bypass filter): git commit, git push, git checkout, git reset, git stash, git merge, git rebase — rm, mv, cp, mkdir, touch, chmod — docker run, docker build — npm install, pip install, brew install, cargo build — any command containing > or | output redirection — sudo, eval, exec.

Secret Detection always on

Every file read and every command output passes through SecretDetector before being returned to the agent. Matched values are replaced with a labeled placeholder — Claude sees that the secret existed but not its value.

Applies to both senkani_read and senkani_exec output — including secrets that appear in shell command output, not just files.

Layer 3 — Smart Denials

Layers 1 and 2 compress what the agent reads. Layer 3 eliminates tool calls that were wasteful by construction — returning a deny at PreToolUse so the underlying tool never fires. Zero tokens consumed, not just reduced.

Five patterns trigger a smart denial. In each case the deny response includes the answer inline — the agent gets the information it needed without the token cost of the tool call.

Re-read Suppression
Fires when a Read targets a path already in the session cache with an unchanged mtime. The file was read within the last 5 minutes and nothing has changed on disk.
DENY: "Already read 42s ago at /src/api/auth.ts (mtime unchanged). Use your existing knowledge of this file."
✓ Tool never executes. 0 tokens consumed. Cached content already in context.
Command Replay
Fires when a Bash call matches a deterministic, read-only command already executed this session with identical arguments and no intervening file changes. Common with git status, ls, test runners run twice.
DENY: "Command `git status` already ran this session (2m ago). Result: On branch feat/order-pagination, 2 files modified."
✓ No subprocess spawned. Prior output returned from session cache.
Symbol Search Upgrade
Fires when a Grep call is detected as a plain-identifier lookup (no regex metacharacters). Instead of running grep across files, the symbol index answers it in ~50 tokens with richer data.
DENY: "Symbol search available for `OrderRepository`. src/orders/repo.ts:12 class OrderRepository src/orders/repo.ts:47 findByUser(userId, limit) Use mcp__senkani__search for full results."
✓ ~50 tokens vs ~5,000 for a grep scan. File:line, kind, and container returned.
Trivial Routing
Fires when a Bash command is deterministically answerable without execution: pwd, whoami, hostname, echo <literal>, date. The hook binary evaluates the command statically.
DENY: "Trivial command resolved: `pwd` → /Users/chris/projects/acme-api"
✓ No subprocess. No disk I/O. Answered from process environment.
Redundant Validation
Fires when the agent issues a Read or Bash cat immediately after a Write or Edit to the same path — a common pattern when an agent reads back what it just wrote to confirm the write succeeded.
DENY: "Redundant read suppressed: /src/orders/repo.ts was just written at 14:32:01. Content confirmed at write time."
✓ PostToolUse hook on the preceding Write marks the path as freshly confirmed.
Deny response format

Smart denials return a structured JSON response to the MCP runtime. The agent receives the reason string as the tool result and continues from there.

{ "hookSpecificOutput": { "hookEventName": "PreToolUse", "permissionDecision": "deny", "permissionDecisionReason": "Already read 42s ago…" } }
⚠ Tier 2 agents: Cursor and Copilot agent mode handle deny responses differently. Some surface the reason string; some treat deny as an empty result. senkani doctor shows how your host handles denials.

Symbol Indexer

Tree-sitter AST extraction across 25 languages. Vendored C parsers, no runtime grammar loading. The same index that intercepts Grep calls also powers senkani_search, senkani_fetch, senkani_explore, and senkani_deps.

Tree-sitter

25 grammars vendored in the binary

AST precision across Swift, TypeScript, TSX, JavaScript, Go, Rust, Python, Java, C, C++, C#, Ruby, PHP, Kotlin, Bash, Lua, Scala, Elixir, Haskell, Zig, HTML, CSS, Dart, TOML, GraphQL.
C parser sources live in Sources/ as vendored submodules. No network fetch, no runtime grammar loading. Binary includes all parsers. senkani grammars shows pinned versions.
Incremental updates

Large projects re-index in <100ms

FSEvents ring buffer watches the project root. File changes emit events; the indexer re-parses only the affected file and replaces its entries atomically.
TreeCache stores (file path → parsed AST). On change: targeted re-parse for files <50MB. If >50 files change simultaneously, full re-index falls back. In-memory IndexEntry array is swapped atomically.
Container hierarchy

Methods are indexed inside their class

Searching findByUser returns OrderRepository.findByUser at line 47 — not just a bare function name. The parent class is always part of the result.
During AST walking the indexer tracks the parent node stack. When a function node is inside a class/struct node, the class name is stored as the container field on the IndexEntry.
Dependency graph

Bidirectional import navigation

senkani_deps answers "what does X import?" and "what imports X?" across 15+ languages. senkani_explore returns a bounded subgraph from any entry point.
DependencyExtractor analyzes import statements via tree-sitter during the indexing pass. Stores directed edges in both directions. Query returns a subgraph, not the full adjacency list.
Symbol fetch

Read only a symbol's lines, not the file

senkani_fetch OrderRepository reads lines 12–94 from repo.ts — not the 800-line file. Token cost is proportional to the symbol, not the file.
The index stores exact startLine and endLine per symbol. senkani_fetch slices those lines. If the file is already in ReadCache, the slice is zero-cost.
Regex fallback

Unsupported languages still indexed

Files in languages without a vendored tree-sitter grammar are parsed by a pattern-matching fallback covering the most common identifier declarations.
RegexBackend applies language-specific patterns to extract function/class-like identifiers. Results have kind "symbol" rather than a precise label; no container hierarchy.

Read Cache

The first time a file is read through senkani_read, the compressed result is stored in a per-session cache. If the file hasn't changed, every subsequent read that session is instant — no disk I/O, no compression, no tokens.

Cache key and invalidation

The cache key is (absolutePath, mtime) — modification time from stat(2). Invalidation is automatic: any mtime change triggers a miss on next access.

Hit path: return stored bytes (already compressed + secrets-scanned). No disk I/O.
Miss path: read from disk → FilterPipeline → SecretDetector → store result + mtime → return.
Scope: single MCPSession. Each agent session gets a fresh session object. Cache is in-memory; not persisted across sessions.
Layer 3 integration: cache hits are also checked by HookRouter — a hit at PreToolUse triggers a re-read suppression DENY rather than routing to senkani_read (the underlying Read never fires).
Clear: senkani_session with action: "clear-cache", or senkani stats --clear-cache.

Budget Enforcement

Daily and weekly cost caps enforced at PreToolUse — before the agent executes any tool, before a token is consumed. If the cap is exceeded, the hook returns a deny decision and the tool never runs.

How it works

On each PreToolUse event, HookRouter checks BudgetConfig (daily + weekly limits in USD) against costForToday() and costForWeek() — real-time SQLite aggregates over the token_events table.

Hard limit exceeded: hook response is DENY with "daily budget $X.XX exceeded" — tool call blocked.
Soft limit (80% of hard): ALLOW + warning appended to tool result so the agent knows budget is nearly depleted.
Runs on the same 5ms hook path. Budget decisions are written to the database for audit. Config lives at .senkani/config.json in the project root.

// .senkani/config.json { "budget": { "dailyLimitUSD": 5.00, // hard daily cap "weeklyLimitUSD": 25.00, // hard weekly cap "softLimitPct": 0.80 // warn at 80% of hard limit } }

Env var overrides: SENKANI_BUDGET_DAILY=5.00 and SENKANI_BUDGET_WEEKLY=25.00 override the config file. Useful for per-project overrides in CI.
Resolution order: env var > config file > no limit (unlimited by default if neither is set).

Terse Mode

The other compression features reduce tokens the AI reads. Terse mode reduces tokens the AI writes. Two layers: a system prompt injection and an algorithmic filler stripper. Per-pane toggle — the T button.

System prompt injection

When T is enabled for a pane, the MCP session prepends this message to every tool result. The AI receives it before processing each response and adjusts its output accordingly.

// Injected before every tool result when T is on CRITICAL: You are in TERSE MODE. ALL responses must minimize output tokens ruthlessly. No preamble, no summaries, no apologies, no "I'll help you with", no "Let me know if you need anything else." Respond only with exactly what was asked. Code blocks pass through unchanged — only prose is compressed.
Algorithmic minimization

A second layer strips filler phrases from tool outputs before forwarding them — removing the noise before it reaches the model, not after. Code blocks are passed through unchanged.

Stripped phrase patterns (case-insensitive): "It looks like", "I'll help you with", "Certainly!", "Of course!", "Let me know if", "I hope this helps", "Feel free to", "Is there anything else", "Happy to help", "I'd be happy to".

Works synergistically with F/C/S/I: those reduce input tokens; T reduces output tokens. Combined effect on a typical session: 40–65% total token reduction vs no compression.

Session Database

Every token saved is measured and persisted. Numbers come from actual byte deltas between raw and compressed output, not estimates. SQLite + FTS5 at ~/Library/Application Support/Senkani/senkani.db.

Schema

sessions — UUID, start/end timestamps, aggregated metrics (rawBytes, savedBytes, commandCount, totalCostUSD)
commands — FTS5-indexed: tool_name, command text, output_preview. Full-text searchable via senkani stats --search <term>
token_events — per-call record: inputTokens, outputTokens, savedTokens, costUSD, source (mcp_tool / hook / intercept), paneId, feature
sandboxed_results — large outputs paginated by r_* IDs for multi-part retrieval

WAL mode for concurrent reads. Budget queries (costForToday(), costForWeek()) are SQLite aggregates on token_events. FTS5 query injection is prevented by stripping operator syntax before queries.

Compliance rate

The analytics pane shows what percentage of tool calls were optimized by Senkani vs passed through unmodified. This surfaces which parts of your workflow still have compression headroom.

compliance = (hook_events + senkani_tool_calls) / total_tool_calls
Result: 0.0–1.0, displayed as "94% of tool calls optimized". A low compliance rate means the agent is using built-in tools directly instead of senkani equivalents — run senkani doctor to check hook registration.

FCSIT toggles

Five per-pane buttons control which compression features are active for that pane's agent session. Each pane can run an independent feature set.

F = Filter pipeline (senkani_exec output compression)
C = Cache (senkani_read session cache)
S = Secrets detection (SecretDetector on all output)
I = Indexer (senkani_search, senkani_fetch active)
T = Terse mode (output token reduction)

Defaults: F/C/S/I on, T off. Persisted to ~/.senkani/panes/{paneId}.env. Changes take effect immediately — no process restart.

Configuration

Resolution order: CLI flag > env var > .senkani/config.json > default. Any toggle can be overridden at any level without touching other settings.

Environment variables

SENKANI_FILTER=on|off    — filter pipeline (default: on)
SENKANI_CACHE=on|off     — session read cache (default: on)
SENKANI_SECRETS=on|off   — secret detection (default: on)
SENKANI_INDEXER=on|off   — symbol indexer (default: on)
SENKANI_TERSE=on|off     — terse mode (default: off)
SENKANI_INTERCEPT=on     — activate hook interception
SENKANI_BUDGET_DAILY=5.00 — daily spend cap in USD
SENKANI_BUDGET_WEEKLY=25.00 — weekly spend cap in USD
SENKANI_PANE_ID=<id>     — MCP activation (injected by app via execve; never set manually)

Dual-mode binary

No flag needed to switch between GUI workspace and MCP server. The binary detects its context at startup via a stdin pipe check.

Piped stdin → MCP JSON-RPC server mode: reads tool calls from stdin, writes results to stdout.
No piped stdin → SwiftUI workspace launches.
MCP activation is gated on SENKANI_PANE_ID — only present in Senkani-managed terminals (injected by app via execve()). Non-Senkani terminals never see senkani tools — zero pollution.

senkani init — what it modifies

Runs once. Idempotent — safe to run again after updates.

1. Writes hook binary to ~/.senkani/bin/senkani-hook (compiled Mach-O, zero deps, <5ms startup).
2. Appends MCP server entry to ~/.claude/settings.json under mcpServers.
3. Appends PreToolUse and PostToolUse hook entries with matcher Read|Bash|Grep|Write|Edit.
4. Does not touch any other settings. Does not require sudo. Does not modify shell profiles.

senkani init --hooks-only skips MCP registration (for Cursor / Copilot users who don't use Claude Code).
