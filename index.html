Hook relay

Tool calls are intercepted
before Claude executes them.

Claude Code exposes a hook system: pre-tool-use and post-tool-use events emitted as JSON to a registered process's stdio. Senkani registers a hook binary that intercepts every Read, Bash, and Grep call and redirects it through the compression pipeline — before the original tool fires.

How
SenkaniHook binary (Sources/HookRelay/) — zero non-Foundation dependencies, ~5ms round-trip latency.

Routing logic:
Read → redirect to mcp__senkani__read (caching, ANSI strip, secret redaction)
Bash → redirect to mcp__senkani__exec if read-only; passthrough for mutating commands (git commit, rm, docker run, npm install) and shell builtins (cd, export, eval, sudo)
Grep → redirect to mcp__senkani__search if pattern is a plain identifier (alphanumeric + underscore); passthrough for regex patterns

Response format: {} (passthrough) or {"hookSpecificOutput": {"permissionDecision": "deny", "reason": "…"}}.
Budget enforcement also runs on this path — if the daily or weekly spend cap is exceeded, the hook returns a deny decision before any token is consumed.

Layer 3 — re-read suppression: if a file was already served by mcp__senkani__read within the last 5 minutes and its mtime hasn't changed, the hook denies the Read call with: "This file was already read 42s ago and hasn't changed. Use your existing knowledge." The tool call never executes. Zero tokens consumed — not redirected, eliminated.

Filter pipeline

Shell output is compressed
before Claude reads it.

senkani_exec runs a command and applies a rule set before returning output. 24 built-in rules cover the noisiest commands in a typical dev workflow — git, npm, cargo, docker, make, yarn, pip, and more. Rules are matched by (command, subcommand) pair.

How
FilterEngine (Sources/Filter/FilterEngine.swift) applies an ordered rule chain. Available operations:

stripANSI — remove all ANSI escape sequences
head(n) / tail(n) — keep first or last n lines
truncateBytes(n) — hard byte cap on output
keepMatching(pattern) — keep only lines matching regex
stripMatching(pattern) — remove lines matching regex
dedupLines — collapse consecutive duplicate lines
groupSimilar — fold runs of similar lines into "… and N more"
stripBlankRuns — collapse multiple blank lines to one

Example: npm install → stripANSI, dedupLines, stripMatching(progress bars), tail(20). 400-line npm output → 8 lines.

Symbol indexer

Find any symbol in ~50 tokens.
Not ~5,000.

When Claude Code calls Grep for an identifier, the hook intercepts it and queries the local symbol index instead. The index stores exact file locations, line ranges, kinds, and container hierarchy for every function, class, struct, and enum in the project.

How
SymbolIndexer (Sources/Indexer/) walks the project on startup using FileWalker, groups files by language, and routes each to either TreeSitterBackend or RegexBackend.

TreeSitterBackend creates an AST from vendored C parsers, walks nodes to extract: function name, class name, struct/enum/protocol, exact start/end line, container (method-inside-class), kind label.

Results land in an in-memory IndexEntry array. SymbolIndex.search() does substring matching on name, kind, file path, and container. Typical query result: 1–5 entries, ~50 tokens. Grep equivalent: full file scan, ~5,000 tokens.

Indexer internals

How the index
actually works.

Tree-sitter

22 grammars, vendored in the binary

AST precision across Swift, Python, TypeScript/TSX/JS, Go, Rust, Java, C, C++, C#, Ruby, PHP, Kotlin, Bash, Lua, Scala, Elixir, Haskell, Zig.
C parser sources live in Sources/ as vendored submodules. No runtime grammar loading, no network fetch. Binary includes all 20 parsers. Grammars are pinned — senkani grammars shows versions.
Incremental updates

Large projects re-index in <100ms

Only changed files are re-parsed. The file watcher emits change events; the indexer re-parses the affected file and updates only its entries in the index.
TreeCache stores (file path → parsed AST). On change: targeted re-parse for files <50MB. If >50 files change simultaneously, full re-index. In-memory IndexEntry array is replaced atomically.
Dependency graph

Navigate by import relationships

senkani_explore and senkani_deps expose a bidirectional import graph. "What does X import?" and "What imports X?" across 15+ languages.
DependencyExtractor analyzes import statements via tree-sitter during the same indexing pass. Stores directed edges in both directions. Query returns a subgraph, not the full adjacency list.
Container hierarchy

Methods are indexed inside their class

A search for findByUser returns the result as OrderRepository.findByUser at line 47, not just a bare function name.
During AST walking, the indexer tracks the parent node stack. When a function node is inside a class/struct node, the class name is stored as the container field on the IndexEntry.
Regex fallback

Unsupported languages still get indexed

Files in languages without a vendored tree-sitter grammar are parsed by a pattern-matching fallback that covers the most common identifier declarations.
RegexBackend applies language-specific patterns to extract function/class-like identifiers. Results have kind "symbol" rather than a precise kind label, and no container hierarchy.
Symbol fetch

Read only a symbol's lines, not the whole file

senkani_fetch OrderRepository reads lines 12–94 from repo.ts — not the 800-line file. Token cost is proportional to the symbol, not the file.
The index stores exact startLine and endLine for each symbol. senkani_fetch slices those lines from disk. If the file is already in the ReadCache, it's a zero-cost slice.
Secret detection

API keys are stripped before
Claude reads them.

Every file read and every command output passes through SecretDetector before being returned to Claude Code. Matched values are replaced with a labeled placeholder — the redaction is visible in the tool result so Claude knows the value existed but was intentionally withheld.

How — 7 regex patterns (Sources/Filter/SecretDetector.swift)
Anthropic API key — sk-ant-[a-zA-Z0-9_-]{20,} → [REDACTED:ANTHROPIC_KEY]
OpenAI API key — sk-[a-zA-Z0-9]{20,} → [REDACTED:OPENAI_KEY]
AWS Access Key ID — AKIA[0-9A-Z]{16} → [REDACTED:AWS_ACCESS_KEY]
AWS Secret Access Key — label + value pattern → [REDACTED:AWS_SECRET]
GitHub Token — gh[pousr]_[A-Za-z0-9_]{36,} → [REDACTED:GITHUB_TOKEN]
Generic API key/secret — (api[_-]?key|api[_-]?secret|auth[_-]?token)[=:][A-Za-z0-9_\-]{20,} → [REDACTED:API_KEY]
Bearer token — (?i)bearer\s+[A-Za-z0-9_\-\.]{20,} → [REDACTED:BEARER_TOKEN]

Applied on both senkani_read and senkani_exec output. Cannot be disabled per-call — only the S toggle on a pane bypasses it for that pane's context.
Read cache

Re-reading an unchanged file
costs zero tokens.

The first time Claude reads a file through senkani_read, the result is compressed and stored in a per-session cache. If the file hasn't changed on disk, every subsequent read in that session returns the cached result instantly — no disk I/O, no compression overhead, no tokens consumed.

How — ReadCache (Sources/Core/ReadCache.swift)
Cache key: (absolutePath, mtime) — modification time from stat(2).

On read: check cache. Hit: return stored bytes (already compressed + secrets-scanned). Miss: read from disk → apply FilterPipeline → run SecretDetector → store result + mtime → return.

Cache is scoped to a single MCPSession. Each Claude Code session gets a fresh session object. Cache is held in memory; not persisted across sessions. Cache invalidation is automatic: mtime change → cache miss on next access.
Budget enforcement

Spending limits that block execution,
not just warn about it.

Daily and weekly cost caps are enforced at the pre-tool-use hook — before Claude Code executes any tool, before a token is consumed. If the cap is exceeded, the hook returns a deny decision and the tool never runs.

How — BudgetConfig + hook path (Sources/Core/BudgetConfig.swift)
On each pre-tool-use event, the hook reads BudgetConfig (daily + weekly limits in USD). It queries the session database for costForToday() and costForWeek() — both are real-time SQLite aggregates over the token_events table.

If either limit is exceeded: hook response is {"hookSpecificOutput": {"permissionDecision": "deny", "reason": "daily budget $X exceeded"}}.

This runs on the same 5ms hook path. Budget decisions are written to the database for audit. Config is in .senkani/config.json at the project root; overridable by env var SENKANI_BUDGET_DAILY / SENKANI_BUDGET_WEEKLY.
Terse mode

Reduce Claude's output length,
not just what it reads.

The other five compression toggles reduce tokens coming in. Terse mode reduces tokens going out. It works in two layers: a system prompt injection that instructs Claude to minimize responses, and an algorithmic input compressor that also strips filler from tool outputs.

How — TerseCompressor (Sources/Filter/TerseCompressor.swift)
System prompt injection: when T is enabled for a pane, the MCP session prepends a system message to every tool result: "CRITICAL: You are in TERSE MODE. ALL responses must minimize output tokens ruthlessly. No preamble, no summaries, no apologies. Respond only with what was asked."

Algorithmic minimization: filler phrases ("It looks like", "I'll help you with", "Certainly!", "Let me know if") are stripped from input before forwarding. Code blocks are passed through unchanged.

Per-pane toggle (T button). Default: off. Works synergistically with F/C/S/I — terse reduces output tokens while the others reduce input tokens.
Local ML

Embeddings and vision.
No API call. No cost.

senkani_embed and senkani_vision run models on Apple Silicon. The embedding infrastructure is also what powers the upcoming semantic search pane — every file in the project will be queryable by meaning, not just text match.

How — MLX models (Sources/MCPServer/)
senkani_embed: MiniLM-L6-v2 via MLX framework. Input: text string. Output: 384-dimensional float vector. Inference on Neural Engine, sub-200ms. Used internally for semantic similarity; also callable as an MCP tool.

senkani_vision: Gemma vision model via MLX. Input: image path or base64 PNG. Output: text description / structured JSON. Handles OCR, screenshot analysis, UI element detection. Sub-500ms on M-series.

Models are downloaded on first use to ~/Library/Application Support/Senkani/models/ via the Model Manager pane. Fully offline after download. No data leaves the machine.
Session database

Every token saved is
measured and persisted.

Senkani tracks what was compressed, how much was saved, and what it cost — per tool call, per pane, per session, lifetime. Numbers come from actual byte deltas between raw and compressed output, not estimates.

How — SessionDatabase (Sources/Core/SessionDatabase.swift)
SQLite + FTS5 at ~/Library/Application Support/Senkani/senkani.db. WAL mode for concurrent reads.

sessions — UUID, start/end timestamps, aggregated metrics (rawBytes, savedBytes, commandCount, totalCost)
commands — FTS5-indexed: tool_name, command text, output_preview. Full-text searchable via senkani stats --search
token_events — per-call record: inputTokens, outputTokens, savedTokens, costUSD, source (mcp/hook/claude), paneId
sandboxed_results — large outputs paginated by r_* IDs for multi-part retrieval

Budget queries: costForToday() and costForWeek() are SQLite aggregates on token_events. FTS5 query injection is prevented by stripping operator syntax before queries.

Configuration

How the pieces
connect and configure.

FCSIT toggles

Per-pane compression profile

Five toggles per pane: F=Filter, C=Cache, S=Secrets, I=Indexer, T=Terse. Each pane runs an independent feature set.
Persisted to ~/.senkani/panes/{paneId}.env. The hook script sources this file on each invocation. Changes take effect immediately — no process restart needed.
Resolution order

CLI flag > env var > config file > default

Any toggle can be overridden at any level without touching other settings.
Env vars: SENKANI_FILTER, SENKANI_CACHE, SENKANI_SECRETS, SENKANI_INDEXER, SENKANI_TERSE. Config file: .senkani/config.json at project root. Defaults: F/C/S/I on, T off.
MCP registration

senkani init wires everything in one command

Reads ~/.claude/settings.json, appends the MCP server entry, injects required env vars. Nothing to configure by hand.
Adds senkani-mcp to mcpServers with stdio transport. Injects SENKANI_PROJECT_ROOT, SENKANI_METRICS_FILE, SENKANI_PANE_ID. Idempotent — running twice doesn't duplicate the entry.
Dual-mode binary

Same binary: GUI workspace or MCP server

No flag needed to switch modes. The binary detects its context at startup and behaves accordingly.
Checks if stdin is a pipe at launch. Piped stdin → MCP JSON-RPC server mode (reads tool calls from stdin, writes results to stdout). No piped stdin → SwiftUI workspace launches.
Workspace persistence

Pane layouts survive restart

Close the app, reopen it, every pane is exactly where it was — same position, same content, same feature toggles.
WorkspaceModel serializes layout to ~/.claude/workspaces/{projectName}.json. Multi-project support: each project has its own file. Backward compatible: no JSON → implicit "Default" project created.
Process lifetime

MCP server exits when Claude Code exits

No zombie MCP processes accumulating in the background. The server monitors its parent and exits cleanly when the parent dies.
Polls getppid(); if parent reparents to launchd (PID 1), the server initiates shutdown. Also: 2-hour session timeout to prevent accumulation. SIGTERM/SIGINT handled with DispatchSource signal handlers for graceful shutdown.
Terminal

A native macOS terminal.
Not a web wrapper.

The terminal pane uses SwiftTerm — a Swift-native terminal emulator. GPU-accelerated rendering on Apple Silicon. Sub-3ms render latency at p50. True color, ligatures, Powerline symbols. The same terminal you're used to, embedded in the workspace.

How — SwiftTerm (Sources/SenkaniApp/)
SwiftTerm handles the VT100/xterm emulation layer. Each terminal pane manages its own PTY via PseudoTerminal — signals are forwarded correctly (SIGWINCH on resize, SIGINT on Ctrl-C).

Render path: character buffer → Metal-accelerated glyph atlas → Core Animation layer. Result: p50 2.1ms, p99 3.4ms render latency measured by senkani bench on M-series hardware.

Process lifecycle: the terminal pane defers startProcess to viewDidMoveToWindow — the PTY is not created until the pane is actually visible, which prevents off-screen processes accumulating on startup.
